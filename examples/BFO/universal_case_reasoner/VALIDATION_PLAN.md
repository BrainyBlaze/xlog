# Validation Plan: BFO Universal Case Reasoner

This plan refines `GOALS.md`, `GQM.md`, and `REQUIREMENTS.md` into the local
validation surface for `examples/BFO/universal_case_reasoner/`. It is
deliberately fail-closed: an unsupported or missing production capability is a
`FAIL` blocker in `validation_summary.json`, not a reason to weaken the target.

## P0 Hard Gates

Authoritative command:

```bash
./validate.sh --strict --gpu-required
```

Validation must record the command line, git branch, git SHA, runtime details,
P0 gate status, GQM metrics, raw evidence paths, explanation paths, and exact
blockers. `validate.sh` delegates to `tools/validate_universal_case_reasoner.py`
and writes `validation_summary.json` by default.

Smoke runners are not production transfer evidence. They may prove that local
surfaces work, but `TRANSFER-002`, `TRANSFER-003`, `NEURAL-002`, and GQM
`Q5`-`Q8` require `evidence/production_transfer.json` with `scope:
"production"`, real Hugging Face dataset source provenance, per-case metric
inputs, computed confusion counts, and the full five-domain holdout protocol.
The provenance check reloads the referenced Hugging Face rows/files, verifies
row and field hashes, and rejects ordinary HF class-label mappings masquerading
as root-cause truth.

Production transfer/profile/soak evidence is generated by:

```bash
python tools/run_production_transfer.py \
  --output evidence/production_transfer.json \
  --latency-samples 30
```

That runner performs the long soak separately from strict validation, then
`./validate.sh --strict --gpu-required` consumes the recorded evidence. The
strict validator still fails closed if the production evidence path is missing,
marked `scope: "development"`, below scale, below soak duration, or missing the
required transfer, ablation, Hugging Face provenance, or bundle-reuse fields.

## BFO Transfer Conformance

Evidence sources:

- `bfo/kernel.xlog`
- `domains/domain_inventory.json`
- Core checksum in `validation_summary.json`

Checks:

- At least 12 BFO upper categories are declared.
- At least 8 BFO relation families are declared.
- Every domain class in every adapter maps to a BFO category.
- All adapters use the same core checksum.
- Adapter/core rule ratio is `<= 0.25` per domain.
- Invalid cross-domain fixtures are rejected or marked inconsistent.

## Domain Coverage

Evidence sources:

- `domains/domain_inventory.json`
- Per-domain case and explanation fixtures added under `domains/`

Checks:

- At least five domains are present.
- Each domain includes root-cause, failure-chain, risk-state, intervention, and
  explanation fixtures.
- The required domain families include clinical, manufacturing, cybersecurity,
  scientific or lab operations, and cloud operations or quality incidents.

## Neural Requirements

Evidence sources:

- XLOG source containing `nn/4`.
- CUDA PyTorch model registration.
- Neural runner raw outputs.
- Baseline and ablation metric records.
- `programs/production_ranker.xlog`.
- `evidence/production_transfer.json`.

Checks:

- A real CUDA PyTorch model is registered through `pyxlog.Program.register_network`.
- The model is invoked by an XLOG `nn/4` declaration.
- Neural outputs materially change root-cause or intervention ranking.
- Neural-only, domain-symbolic, shared-symbolic, and neuro-symbolic ablations
  are all measured.
- Neuro-symbolic uplift over the strongest baseline is at least 15%.
- Fixed-seed outputs are byte-identical across five runs.

## Transfer And Evolution

Evidence sources:

- Holdout protocol metadata in `domains/domain_inventory.json`.
- Hugging Face dataset source records in `evidence/production_transfer.json`.
- Rule evolution logs.
- Core mutation audit.
- Per-domain metric files.
- `evidence/production_transfer.json`.

Checks:

- At least one domain is withheld during rule evolution.
- The held-out domain is not used while evolving the shared BFO rules.
- The same BFO kernel is evaluated over all five domains with no core edits.
- Production metrics are recomputed from `metric_inputs.prediction_records`,
  `metric_inputs.ablation_records`, and invalid fixture records.
- Held-out root-cause F1 is at least `0.90`.
- Accepted intervention precision is at least `0.95`.
- Non-held-out promoted rule quality is precision `>= 0.98`, recall `>= 0.95`,
  and F1 `>= 0.965`.
- Learned rules do not mutate `bfo/kernel.xlog`.
- Every top-level claim has a BFO-valid explanation path.

## DILP Rule Induction

Evidence sources:

- `programs/dilp_proof_paths.xlog`.
- `dilp_report` in `evidence/production_transfer.json`.
- `metric_inputs.dilp_prediction_records` in
  `evidence/production_transfer.json`.

Checks:

- XLOG proof-path clauses execute and provide proof candidates for learned rule
  selection.
- Neural predicate parameters and symbolic rule weights train jointly on CUDA.
- Learned rule inventories cover every leave-one-domain-out fold and record
  selected clauses, clause weights, training domains, and held-out exclusion.
- Clause ablations are recomputed from raw DILP prediction records, and the
  full model reaches macro F1 `>= 0.90` while matching or beating every
  single-clause ablation.
- Proof-level support tensors stay device-resident and receive nonzero
  gradients.
- Held-out cases and labels are excluded from rule induction; held-out labels
  are used only after prediction for metric recomputation.

## Robust Generalization

Evidence sources:

- `metric_inputs.generalization_prediction_records` in
  `evidence/production_transfer.json`.
- `generalization_report` in `evidence/production_transfer.json`.
- Per-record Hugging Face row, root-field, and root-text hashes.
- Freeze manifest for BFO rules, learned rules, neural architecture,
  thresholds, aliases, and scoring weights.
- Generalization ablation records and adversarial domain-shift records.

Checks:

- Leave-one-domain-out evaluation covers every production domain.
- Each held-out domain contributes at least 100 real Hugging Face cases with
  row and field hashes.
- Macro held-out root-cause F1 is at least `0.90`, and the minimum per-domain
  F1 is at least `0.85`.
- Candidate spaces are built from frozen catalogs or training-domain evidence,
  not from held-out test RCA/root/intervention labels.
- Model/rule/threshold/alias/scoring freeze metadata is recorded before
  held-out evaluation.
- At least one held-out evaluation uses an unseen dataset family.
- Strong baselines include neural-only, symbolic-only, domain-specific
  classifier, retrieval/RAG nearest-neighbor, majority/prior, and
  neuro-symbolic methods.
- Bootstrap confidence intervals and paired significance tests are reported.
- Noisy, sparse, paraphrased, missing-field, and distractor-candidate variants
  are evaluated.
- The validator recomputes aggregate metrics from raw generalization records
  and fails if any production domain is excluded.

## Device-Resident Execution

Evidence sources:

- `pyxlog` host-transfer stats.
- D2H/H2D transfer counters captured before and after the hot loop.
- CUDA tensor checks from the neural and symbolic runners.

Checks:

- Data-plane D2H transfers in the hot loop are `0`.
- Data-plane H2D transfers after initial load are `0`.
- Control-plane metadata per hot iteration is `<= 4096` bytes.
- Semantic column downloads or host tensor materialization in the hot loop are
  immediate failures.

## Scale And Performance

Evidence sources:

- Production profile output.
- Soak-run output.
- Latency histogram and memory telemetry in `validation_summary.json`.
- `evidence/production_transfer.json`.

Checks:

- Symbolic/BFO facts `>= 1,000,000`.
- Neural observations `>= 100,000`.
- Entities `>= 50,000`.
- Staged delta updates `>= 10,000`.
- p95 core indexed query latency `<= 50 ms`.
- Strict validation wall time `<= 180 seconds` unless a separate production
  profile justifies a longer execution.
- Soak run `>= 30 minutes`, GPU memory drift `<= 2%`, and no unbounded relation
  growth.

## Evidence Schema

`validation_summary.json` must include:

- Branch and git SHA.
- Hardware and CUDA/PyTorch/XLOG runtime details.
- Commands executed.
- Metric values for every GQM question.
- PASS/FAIL for every P0 gate.
- Raw output paths.
- Explanation paths.
- Failure/blocker descriptions with exact requirement IDs.

## XLOG Findings Output

The production report is incomplete if it only reports PASS metrics, soak time,
and latency. The project must also emit an XLOG findings bundle:

- `XLOG_FINDINGS.md`: human-readable answers to what the project revealed about
  XLOG, what broke or required workaround code, what feature is missing or weak,
  what belongs in core XLOG versus pyxlog versus the example, and which minimal
  upstream tests should be added.
- `xlog_issue_ledger.json`: machine-readable findings with component,
  severity, expected behavior, actual behavior, minimal reproducer command,
  evidence artifact path, proposed fix, and suggested regression test.
- `repro/`: one minimal XLOG or Python reproducer per issue.
- `proposed_fixes.md`: concrete XLOG, pyxlog, and example improvements, with
  regression test candidates mapped to repo locations.

This output is especially important for the DILP boundary: UCR now includes the
v0.8.9 upstream XLOG/pyxlog surfaces for joint symbolic/neural rule-weight
learning, learned clause inventories, proof-level gradients, CUDA audit
contracts, module-boundary diagnostics, and grouped transfer metrics.
