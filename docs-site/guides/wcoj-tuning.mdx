---

Audience: downstream users who need to decide whether to enable, force, tune,
or debug xlog WCOJ routes.

This guide is about operational choices. It does not restate the kernel
architecture. It answers:

- Does my rule shape route through WCOJ?
- What makes it fall back?
- Which env vars or config builders should I use?
- Should I choose Cardinality, Skew, LeaderCardinality, or HeatAware?
- Which tuning move is appropriate for skewed graphs, deep joins, multi-rule
  strata, or recursive workloads?

## Quick Decision Flowchart

```text
Start
 |
 |-- Is the rule positive relational Datalog?
 |      no  -> WCOJ will not route. Use the normal fallback path.
 |      yes
 |
 |-- Does the body match a certified WCOJ family?
 |      triangle: e(X,Y), e(Y,Z), e(X,Z)
 |      4-cycle: e(W,X), e(X,Y), e(Y,Z), e(Z,W)
 |      K=5/K=6 clique: every C(K,2) edge is present
 |      no  -> ordinary binary/hash plan, or follow-on ChainJoin for 2-atom chains
 |      yes
 |
 |-- Are join-key types supported?
 |      supported: U32, U64, Symbol
 |      no -> fallback
 |      yes
 |
 |-- Are all runtime inputs present, same width class, and layout-buildable?
 |      no -> fallback
 |      yes
 |
 |-- Is dispatch disabled or explicitly forced off?
 |      yes -> fallback
 |      no
 |
 |-- Is force dispatch enabled?
 |      yes -> run WCOJ and compare counters/rows
 |      no
 |
 |-- Is adaptive/cost-model dispatch enabled for this shape?
 |      no -> fallback
 |      yes
 |
 |-- Does the selected cost model predict WCOJ should win?
 |      yes -> run WCOJ
 |      no  -> planned hash/fallback
```

The important point is that "eligible" and "dispatched" are different. A rule
can be structurally eligible, promoted into `MultiWayJoin`, and still use the
embedded fallback because a gate is off, stats are incomplete, the cost model
predicts hash wins, or runtime validation declines.

## Eligibility Decision Flow

WCOJ routes are certified by shape, type, and runtime validation.

| Rule family | Current user-facing status | Notes |
|---|---|---|
| Triangle | Production WCOJ route | Supports `U32`, `Symbol`, and `U64` width classes. Triangle adaptive is default-on through `RuntimeConfig::default()` in this branch. |
| 4-cycle | Production WCOJ route, adaptive opt-in | Force gate is available. Adaptive/cost-model mode is off unless enabled by config or env. |
| K=5/K=6 clique | Production WCOJ route after planner cost gate | Requires complete clique edge set and usable stats. May emit a planned hash route instead of WCOJ. |
| K=7/K=8 clique | Follow-on target, not a current user promise on this branch | The planner data structures admit K up to 8, but template, counter, and certification closure are separate. |
| 2-atom chain | Follow-on `ChainJoin`, not WCOJ | Hot two-atom chains route through a chain dispatcher after chain integration. It is not paper WCOJ. |
| Arbitrary deep join tree | Not automatically WCOJ | Use helper-split/K-clique paths where applicable; otherwise fallback is expected. |

Hard fallback reasons include:

- Free Join order planner declines a large bad-order body. Free Join cannot
  reorder a chain to start from a selective tail, so on a large join whose only
  prefix-key-joinable order would materialize a much larger intermediate than
  the binary plan, the planner declines to the binary fallback rather than
  dispatch the loss. Small joins and competitively-ordered bodies are unaffected.
- Ground facts, negation, aggregation boundaries, `is` expressions, or too few
  positive atoms.
- Unsupported join-key types. WCOJ join keys are `U32`, `U64`, and `Symbol`.
- Mixed width classes inside one dispatch.
- Malformed or non-scan `MultiWayJoin` inputs.
- Projection or `slot_vars` not matching the certified shape.
- Missing relation buffers in the runtime store.
- Runtime/provider errors during layout, metadata, count, scan, total, or
  materialize.
- Explicit force-off or disable config.
- Cost model declines.
- For K-clique, incomplete stats or predicted hash win. This is a planned hash
  route, not a promoter miss.

## Fallback Paths

Every promoted `MultiWayJoin` carries an embedded `fallback` tree. That tree is
the post-optimizer binary/hash plan that would have run without WCOJ promotion.
If the dispatcher returns `Ok(None)`, the executor descends into that fallback.

Fallback is not an error. It is the safety boundary that lets you run with
WCOJ enabled in production while preserving row-set semantics.

Use fallback intentionally when:

- You are verifying a regression by comparing forced WCOJ to forced-off output.
- Your workload is small enough that launch/layout overhead is more expensive
  than the avoided binary intermediate.
- Stats are missing and you do not want forced dispatch.
- You are debugging a suspected layout, type, or projection issue.

Do not add new kill switches for one-off tests. Prefer the existing
`RuntimeConfig` builders or env vars listed below.

## Env Vars

Env vars are process-global. In tests, serialize them or prefer per-runtime
builders. Production services should set them once at process startup.

| Env var | Values | Scope | Effect |
|---|---|---|---|
| `XLOG_USE_WCOJ_TRIANGLE_U32` | `1`/`true` enables | Triangle force gate | Forces recognized triangle dispatch. `0`, `false`, unset, or empty do not force. |
| `XLOG_USE_WCOJ_4CYCLE` | `1`/`true` enables | 4-cycle force gate | Forces recognized 4-cycle dispatch. |
| `XLOG_USE_WCOJ_4CYCLE_ADAPTIVE` | `1`/`true` enables | 4-cycle adaptive opt-in | Lets the runtime cost model decide 4-cycle dispatch. Default is off. |
| `XLOG_DISABLE_WCOJ_4CYCLE` | `1`/`true` disables | 4-cycle kill switch | Beats force and adaptive. |
| `XLOG_WCOJ_COST_MODEL` | `cardinality`, `skew`, `skewclassifier` | Runtime cost model | Selects `CostModelKind`. Invalid non-empty values resolve to `SkewClassifier`. Unset defaults to `Cardinality`. |
| `XLOG_WCOJ_BLOCK_WORK_UNIT` | integer `1..8192` | HG block-slice work unit | Default `1024`. Invalid values fall back to default with a warning. |
| `XLOG_DISABLE_FREE_JOIN` | `1`/`true` disables | Free Join kill switch | Forces general multiway bodies through the binary fallback instead of the Free Join engine. |
| `XLOG_DISABLE_WCOJ_GROUPBY_FUSION` | `1`/`true` disables | Aggregate-fused WCOJ kill switch | Forces count/sum/min/max-by-root over a triangle body to materialize then group, instead of the fused aggregate. |
| `XLOG_DISABLE_FACTORIZED_DELTA` | `1`/`true` disables | Factorized recursive-delta kill switch | Forces every semi-naive delta step through the legacy hash-join → diff path. |
| `XLOG_FACTORIZED_DELTA_MAX_DOMAIN` | integer | Factorized-delta dense-domain cap | Largest dense domain the bitvector delta route accepts (default `2^14`, hard bound `2^16`); above it the sparse route or legacy path runs. |

Current branch caveat: triangle adaptive and triangle hard-disable are runtime
config builders, not env resolvers. Use
`with_wcoj_triangle_dispatch_adaptive(...)` and
`with_wcoj_triangle_dispatch_disabled(...)` for those controls.

## Runtime Config Builders

Use `RuntimeConfig` when one process needs different WCOJ behavior for
different executors or tests.

```rust
use xlog_core::{CostModelKind, RuntimeConfig};

// Production default: deterministic execution, triangle adaptive on,
// 4-cycle adaptive off, Cardinality runtime cost model.
let default_runtime = RuntimeConfig::default();

// Force triangle WCOJ for an A/B row-set comparison.
let force_triangle =
    RuntimeConfig::default().with_wcoj_triangle_dispatch(Some(true));

// Force triangle off for the same comparison.
let triangle_off =
    RuntimeConfig::default().with_wcoj_triangle_dispatch(Some(false));

// Runtime-local triangle hard stop. This beats force/adaptive.
let triangle_disabled =
    RuntimeConfig::default().with_wcoj_triangle_dispatch_disabled(Some(true));

// Enable 4-cycle adaptive/cost-model routing.
let adaptive_4cycle =
    RuntimeConfig::default().with_wcoj_4cycle_dispatch_adaptive(Some(true));

// Select the runtime dispatch model explicitly.
let cardinality =
    RuntimeConfig::default().with_wcoj_cost_model(Some(CostModelKind::Cardinality));
let conservative =
    RuntimeConfig::default().with_wcoj_cost_model(Some(CostModelKind::SkewClassifier));
```

Attach the config when constructing the executor:

```rust
use std::sync::Arc;
use xlog_core::RuntimeConfig;
use xlog_runtime::Executor;

let config = RuntimeConfig::default().with_wcoj_triangle_dispatch(Some(true));
let mut executor = Executor::new_with_config(Arc::clone(&provider), config);
```

## CompilerConfig Builders

`RuntimeConfig` decides whether a recognized runtime shape dispatches.
`CompilerConfig` decides whether the compiler emits non-default variable-order
metadata for triangle and 4-cycle plans.

Default compile behavior preserves the baseline:

```rust
let plan = compiler.compile_with_stats_snapshot(source, Some(&stats_snapshot))?;
```

Enable a compile-time variable-ordering model when stats are meaningful:

```rust
use xlog_logic::compiler_config::{CompilerConfig, WcojVarOrderingKind};

let config = CompilerConfig {
    wcoj_variable_ordering: WcojVarOrderingKind::HeatAware,
    wcoj_var_ordering_threshold: 0.5,
};

let plan =
    compiler.compile_with_config_and_stats_snapshot(source, &config, Some(&stats_snapshot))?;
```

`wcoj_var_ordering_threshold` is a ratio gate. The model chooses a non-default
leader only when:

```text
candidate_score / default_leader_score <= threshold
```

Default threshold is `0.5`. Smaller values require a clearer win before
rotating the leader. Values outside `(0.0, 1.0]`, NaN, and infinity clamp back
to the default.

## Cost-Model Trade-Offs

There are two cost-model layers.

Runtime dispatch:

- `CostModelKind::Cardinality` is the default. Use it when relation
  cardinalities and observed selectivity are populated or when you want the
  current production route.
- `CostModelKind::SkewClassifier` is conservative on the current skew-classifier integration code
  path. It acts as a legacy opt-out from stats/cardinality dispatch when you
  need to prove fallback behavior.

Compile-time variable ordering:

- `WcojVarOrderingKind::Disabled` is the default. Use it when you want
  bit-identical leader order to the original WCOJ slices.
- `LeaderCardinality` is the simplest useful model. It picks the smallest
  relation as leader if the threshold gate shows a clear win.
- `HeatAware` combines cardinality, relation heat, and observed selectivity.
  Use it for skewed graphs or repeated workloads where `StatsManager` has
  enough evidence to identify hot relations and tight edges.

Practical selection:

| Situation | Runtime model | Compiler model |
|---|---|---|
| First run, little stats | `Cardinality`, or force only for experiments | `Disabled` |
| Stable batch with seeded cardinalities | `Cardinality` | `LeaderCardinality` |
| Repeated skewed workload with heat/selectivity evidence | `Cardinality` | `HeatAware` |
| Debug fallback or bisect | `SkewClassifier` or explicit force-off | `Disabled` |
| K-clique with incomplete stats | Planned hash is expected | Seed complete stats first |

## Threshold Tuning

Tune one knob at a time and record counters plus row-set equality.

`wcoj_var_ordering_threshold`:

- Lower it, for example `0.25`, when leader rotation is too eager and layout
  overhead dominates.
- Raise it toward `0.75` only after row parity is stable and profiling shows
  the default leader is repeatedly expensive.
- Keep the default `0.5` when you do not have repeatable stats evidence.

`XLOG_WCOJ_BLOCK_WORK_UNIT`:

- Lower it for severe leader-key skew when a few keys dominate work.
- Raise it for uniform inputs when launch/block overhead dominates.
- Stay within `1..8192`. The default is `1024`.
- Validate with dispatch counters and elapsed time; this knob should not
  change row sets.

Force gates:

- Use force gates to prove the WCOJ route can produce the same rows.
- Do not leave force gates enabled as a substitute for a cost model unless the
  workload is fixed and benchmarked.

## Recipe: Skewed Graphs

Symptoms:

- A few root keys dominate runtime.
- Binary fallback materializes large intermediates.
- WCOJ dispatch works but is inconsistent across similar input sizes.

Actions:

1. Seed stats snapshots with relation cardinalities and selectivity if
   available.
2. Compile with `WcojVarOrderingKind::HeatAware`.
3. Keep runtime model at `Cardinality`.
4. Lower `XLOG_WCOJ_BLOCK_WORK_UNIT` one step and remeasure.
5. For K-clique, confirm helper-split specs appear only for buried skew, not
   uniform input.
6. Compare forced WCOJ, default, and forced-off row sets before trusting speed.

Expected evidence:

- WCOJ dispatch counter advances on forced and accepted adaptive runs.
- Row sets match forced-off fallback.
- Hot relation is not selected as leader when `HeatAware` has evidence.

## Recipe: Deep Join Trees

Symptoms:

- Rules contain five or more body atoms.
- Binary fallback OOMs or creates very large intermediates.
- Skew is buried below the first join level.

Actions:

1. If the rule is a complete K=5 or K=6 clique, seed complete K-clique stats
   and let the planner choose WCOJ or planned hash.
2. If the rule has buried skew, rely on helper-split rather than manually
   decomposing the whole rule.
3. If helper-split produces a two-atom helper consumer, use the
   `ChainJoin` route after integration.
4. Do not assume arbitrary deep trees are WCOJ just because they are expensive.
   Unsupported shapes should fall back until a certified promoter exists.

Expected evidence:

- K-clique route is `WcojWithPlan` or `PlannedHashRoute`, not an unstructured
  miss.
- Helper relation count is zero for uniform input and positive for buried
  skew fixtures.
- Fallback parity remains exact.

## Recipe: Multi-Rule Strata

Symptoms:

- Individual rules are too small to saturate the GPU.
- Later fixpoint iterations have small deltas.
- Kernel launch overhead becomes visible.

Actions:

1. Do not force WCOJ for every rule. Let the cost model skip rules that are too
   small.
2. Keep rules independent within a monotonic stratum so stream-mux can batch
   phase-aligned work after integration.
3. Preserve the recorded-launch stream discipline. Avoid ad hoc host syncs.
4. For follow-on integration, validate per-stream pool sizing before increasing
   stream count.

Expected evidence:

- WCOJ counters advance only on rules whose cost model accepts them.
- No stream-pool exhaustion under repeated executor calls.
- Row sets are independent of rule execution order.

## Recipe: Recursive Workloads

Symptoms:

- WCOJ routes fire during seed but not later iterations, or counters are hard
  to interpret.
- Recursive SCCs mix binary fallback and WCOJ routes.
- K-clique runtime histogram needs current per-iteration state.

Actions:

1. Use body-keyed dispatch through the normal executor. Do not call provider
   kernels directly from recursive code.
2. Interpret counters per successful dispatch, not per source rule. A rule can
   dispatch once in seed and once per non-empty delta variant per iteration.
3. Use force-off runs for row-set parity.
4. For K-clique, rely on the Merge-phase metadata refresh accounting; metadata
   is rebuilt from live buffers at launch.
5. Keep `CompilerConfig::default()` if variable-ordering stats are stale.

Expected evidence:

- Forced-off and default/forced WCOJ produce the same final relation.
- Dispatch counters align with seed plus variant count.
- `kclique_histogram_refresh_count()` advances when recursive K-clique rules
  mention a predicate that was merged.

## Debug Checklist

When WCOJ did not run:

1. Check the source shape. Is it triangle, 4-cycle, or K=5/K=6 clique?
2. Check types. Are join keys `U32`, `U64`, or `Symbol`?
3. Check env/config. Did you force off or disable the route?
4. Check stats. Does the cost model have enough cardinality/selectivity?
5. Check counters. Did the route fire but output zero rows?
6. Check fallback parity. Does forced-off output match expected rows?

When WCOJ ran but was slower:

1. Confirm the input is large enough to amortize layout and launch overhead.
2. Try `LeaderCardinality` or `HeatAware` only with meaningful stats.
3. Tune `XLOG_WCOJ_BLOCK_WORK_UNIT` in one direction at a time.
4. For 4-cycle, verify adaptive mode was intentionally enabled.
5. For K-clique, inspect whether the planner predicted hash but force was used
   anyway.

When rows differ:

1. Treat it as a correctness bug.
2. Capture forced-off fallback rows and forced WCOJ rows.
3. Record the RIR shape and projection.
4. Stop tuning until row parity is restored.

## Minimal Evidence Template

For a tuning change, record:

```text
source shape:
runtime config:
compiler config:
env:
stats snapshot source:
fallback rows:
WCOJ rows:
dispatch counters:
elapsed fallback:
elapsed WCOJ:
block work unit:
notes:
```

This makes tuning reproducible and keeps performance claims separate from
correctness claims.
