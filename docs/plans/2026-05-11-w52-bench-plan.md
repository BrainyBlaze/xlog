### Pivot-Heavy Multi-Way Workload

There is no separate pivot-heavy kernel, provider entry, or benchmark in the current tree. This plan defines pivot-heavy as a synthetic K5 data-distribution workload over the existing clique5 provider/runtime/kernel path cited above:

```text
pivot5(P, A, B, C, D) :-
    pa(P, A), pb(P, B), pc(P, C), pd(P, D),
    ab(A, B), ac(A, C), ad(A, D),
    bc(B, C), bd(B, D), cd(C, D).
```

Fixture definition:

- Incident pivot edges `pa/pb/pc/pd`: one heavy pivot key `P=0` with fanout `N`.
- Leaf filter edges `ab/ac/ad/bc/bd/cd`: diagonal filter rows `(i, i)` for `i in 1..=N`.
- Expected output: `N` rows `(0, i, i, i, i)`.
- Binary hash baseline stress: a left-deep chain that joins pivot incident edges first can grow to `N^4` before leaf filters collapse it.
- Cell cap: use `N in {10, 20, 30, 40}` unless implementation recon proves a lower-memory baseline order. `40^4 = 2,560,000`, which stays below the 4,000,000 Cartesian guardrail used in W4.2/W4.3 threshold reasoning.

This synthetic shape is paper-relevant only as a skew stressor. It does not claim P3 histogram-guided scheduling.
