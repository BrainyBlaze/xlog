## Merge posture for the allocator PR

The original v0.6 plan listed A4 as a hard gate worded roughly
"prototype join becomes stable after allocator." That wording is
now falsified. Two posture options:

### Option 1 â€” keep original A4 wording as a hard gate

Allocator PR is **not merge-ready**. The join prototype must be
debugged and stabilized first; the allocator branch waits.

### Option 2 â€” rescope the allocator PR (recommended)

Scope = **v0.6 runtime allocator + opt-in provider/integration
path + A3 stability.** A4 is recorded as:

> "Falsification result. The GPU-resident binary-join prototype
> remains broken on top of the allocator with identical failure
> signatures in both legacy and runtime fixture modes (6/20 vs
> 7/20 pass rate at 8 threads). Allocator is **not** the root
> cause of the prototype corruption. The prototype remains blocked
> and must be debugged separately on `a4-rebase-readiness`."

Under this rescoping the allocator PR is **mergeable** on its own
merits: A3 PASSes; legacy default behavior is preserved; the
runtime fixture is opt-in; `CudaKernelProvider::new` default is
unchanged.

This document recommends Option 2.

## Next action (separate from the allocator PR)

Debugging the join prototype starts from the concrete invariant
violation:

> `compact_buffer_by_device_mask_device_count: \
>  mask len NNN > row cap NN`

Trace which path produces a buffer whose mask length is derived
from one row-domain (the join's full output capacity) while the
compaction input is sized to a smaller row count (the realized
output count or an upstream's `num_rows()`). Likely culprits:

* Deferred status / `try_consume_overflow_status` not propagating
  the realized COO row count back to the buffer's `num_rows()`.
* Downstream operators (`filter`, `compact_*`, etc.) reading
  `num_rows()` from a buffer whose `row_cap` was set on an earlier
  pass and not refreshed after countâ†’materialize truncation.
* Kernel-level race in the new count/probe/materialize chain
  whose effects are masked at single-test-thread but surface
  under `--test-threads=8`.

Reproduce with one failing test in isolation, repeated under
`--test-threads=8 --test=test_network_connectivity`, then bisect
within the prototype commit's relational.rs / join.cu changes.

This work is **out of scope** for the allocator PR.
