# v0.6 Stream/Lifetime-Safety — Prerequisite for GPU-Resident Binary-Join Prototype

**Date:** 2026-04-28
**Branch:** `a4-rebase-readiness` (local; not pushed)
**Allocator branch (already merged to main):** `feat/v0.6-device-runtime-allocator` → `main` @ `2fde633f`
**Prototype commit retained for reference:** `88e49be7` (cherry-picked from `feat/v0.5.5-gpu-resident-binary-join`)

This document records the verdict from the A4 acceptance gate
re-run on top of the merged v0.6 allocator, the deep-research root
cause, and the **prerequisite stream/lifetime-safety PR** that
must land before any further attempt to merge the GPU-resident
binary-join prototype.

This is a planning artifact. No code is being changed by this
commit. No allocator change. No kernel patch. No test mutex.

## A4 verdict

**The GPU-resident binary-join prototype is not mergeable.**

Three-mode 20× `cargo test -p xlog-integration --test real_world_tests --release -- --test-threads=8`:

| mode (production stack: `GlobalDeviceBudget(LoggingResource(AsyncCudaResource))`) | parallel pass | parallel fail | serial (`--test-threads=1`) |
|---|---:|---:|:---:|
| Legacy fixture (`GpuMemoryManager::new` per test, cudarc default allocator)       |  6/20 | 14/20 | 10/10 |
| Per-test runtime fixture (each test → `XlogDeviceRuntime::with_resource`)          |  6/20 | 14/20 |  10/10 |
| Process-wide shared runtime fixture (one `XlogDeviceRuntime` for all tests via `OnceLock`) |  **0/20** | **20/20** |   5/5 |

Single-test loops (`test_social_*`, `test_forward_*`, `test_network_*`, `test_rbac_*`, each ×10 alone, `--test-threads=1`): 40/40 PASS.

Failure signatures across modes are identical: `mask len NNN > row cap NN`, arithmetic-op rows containing values from another query (e.g. `(15, 109, 10)` instead of `(15, 5, 10)`), missing Datalog derivations, aggregate single-iteration failures of 5–7 of 13 tests. Shared mode escalates to outright SIGSEGV.

The allocator is **not** the root cause — even the production stack with one process-wide owner of all device allocations cannot rescue parallel correctness.

## Root cause: xlog mixes three incompatible CUDA models

xlog's GPU layer currently composes three models that are mutually
incompatible without explicit cross-stream synchronization:

1. **CUDA stream-ordered allocation.** `cuMemAllocAsync` /
   `cuMemFreeAsync` require allocation, kernel use, and free to be
   ordered on the same stream — or coordinated via explicit events
   — for memory reuse to be safe. NVIDIA documents this directly
   for `cudaMallocAsync` / `cudaFreeAsync`. RAPIDS RMM's
   `device_memory_resource` shape (which the v0.6 allocator
   mirrors) builds on this guarantee.
2. **cudarc's safe `LaunchArgs` model.** cudarc records read /
   write events on `CudaSlice` and waits on those events before
   freeing the slice. This makes "drop the slice while a kernel is
   queued" sound when launches go through the typed-args API.
3. **xlog's raw-launch shim.** `crates/xlog-cuda/src/cuda_compat.rs:218`
   provides `LaunchAsync<&mut [*mut c_void]>` for `CudaFunction`,
   which dispatches `launch_raw(cfg, params)`. Every launch site
   that builds a `Vec<*mut c_void>` of kernel parameters bypasses
   cudarc's typed event bookkeeping. There is no recorded
   read / write event on the underlying `CudaSlice` instances; the
   slices have no idea a kernel may still be reading/writing them
   when they drop.

The prototype's `count → scan → materialize` join pipeline
amplifies this. Each binary join allocates a count buffer, prefix
buffer, mask, block-sums scratch, output count scalar, output
columns, and a row-count scalar — a half-dozen short-lived
allocations per join, all wired into kernels via the raw-launch
shim. Under multi-thread Executor stress, the rate of
allocate / queue-launch / drop / reallocate cycles explodes, and
the latent bug (no event on the launch, free queued anyway) goes
from "rare" to "majority of iterations corrupt."

## Specific code sites

These are the concrete places the safety contract is broken on
`a4-rebase-readiness`. **The first three (sites 1–3) exist on the
merged main as well — they are mainline stream/lifetime safety
issues that need to be fixed regardless of the prototype.** The
fourth (site 4, `relational.rs:3075/3145`) is the
prototype-specific amplifier — those exact raw-launch sites are
introduced by `88e49be7`. Mainline has its own raw-`Vec<*mut c_void>`
launch sites in other operators that share the same shape, but
the prototype's count→scan→materialize chain creates many more of
them per query.

### 1. Raw-launch shim drops cudarc's event recording

`crates/xlog-cuda/src/cuda_compat.rs:218`:

```rust
unsafe impl LaunchAsync<&mut [*mut c_void]> for CudaFunction {
    unsafe fn launch(
        self,
        cfg: LaunchConfig,
        params: &mut [*mut c_void],
    ) -> Result<(), DriverError> {
        self.launch_raw(cfg, params)
    }
}
```

The `params` slice is opaque pointers; the impl has no way to
know which `CudaSlice<T>` they came from, so it cannot record
the read/write events cudarc's safe path normally records.
Every callsite that constructs `Vec<*mut c_void>` and calls
this `launch` is an event-less launch.

### 2. Mutable-args storage forgets the sync

`crates/xlog-cuda/src/cuda_compat.rs:136`:

```rust
impl<T> IntoKernelParamStorage for &mut CudaSlice<T> {
    type Storage = DeviceParamStorage<'static>;
    fn into_kernel_param_storage(self) -> Self::Storage {
        let stream = self.stream().clone();
        let (ptr, sync) = cudarc::driver::DevicePtrMut::device_ptr_mut(self, &stream);
        std::mem::forget(sync);          // ← cudarc's "wait before free" handle dropped
        DeviceParamStorage::unsynced(ptr)
    }
}
```

`device_ptr_mut` returns a `SyncOnDrop` handle whose `Drop`
records the write event so cudarc's free can wait. `forget(sync)`
discards that handle — the write is never visible to cudarc's
event machinery, so a subsequent slice drop frees memory that may
still be queued for write by a launched kernel.

### 3. Runtime-backed `TrackedCudaSlice` deallocates on the wrong stream

`crates/xlog-cuda/src/memory.rs:414` (allocate side):

```rust
let block = match runtime.allocate(bytes_usize, StreamId::DEFAULT, AllocTag::UNTAGGED) {
    Ok(b) => b,
    ...
};
```

Allocation is bound to `StreamId::DEFAULT`. The cudarc typed view
is built via `upgrade_device_ptr::<T>` over the runtime's raw
pointer.

`crates/xlog-cuda/src/memory.rs:251` (drop side):

```rust
impl<T: cudarc::driver::DeviceRepr> Drop for TrackedCudaSlice<T> {
    fn drop(&mut self) {
        match &mut self.backing {
            Backing::Cudarc => {
                unsafe { ManuallyDrop::drop(&mut self.inner) };
            }
            Backing::Runtime { runtime, block } => {
                if let Some(block) = block.take() {
                    let _ = runtime.deallocate(block);   // ← frees on alloc_stream
                }
                // typed view forgotten; cudarc's events on it never fire
            }
        }
        ...
    }
}
```

`Backing::Runtime` deallocates through the runtime, which queues
`cuMemFreeAsync` on `block.alloc_stream`. The typed `CudaSlice<T>`
view that the launch used is `mem::forget`-ed, so any read/write
events cudarc *might* have recorded on it never get a chance to
fire. The free is ordered on `alloc_stream` only, with no
guarantee that the launch stream (often the device default
stream) has completed any in-flight work referring to this
memory.

### 4. Prototype amplifies the gap with raw `Vec<*mut c_void>` launches

`crates/xlog-cuda/src/provider/relational.rs:3075` and
`crates/xlog-cuda/src/provider/relational.rs:3145` (and many
similar sites):

```rust
unsafe {
    let mut params: Vec<*mut c_void> = vec![
        (&probe.left_packed.hashes).as_kernel_param(),
        probe.left.num_rows_device().as_kernel_param(),
        (&probe_cap).as_kernel_param(),
        (&probe.table.bucket_offsets).as_kernel_param(),
        (&probe.table.bucket_counts).as_kernel_param(),
        ...
    ];
    func.launch(cfg, &mut params)?;   // ← no event recording on any of those slices
}
```

Each of the `as_kernel_param()` calls returns a raw `*mut c_void`
that points at the `CUdeviceptr` field of a `CudaSlice<T>` (or
similar). The slices themselves are not visible to the
launch — cudarc cannot know they're being read or written — so
its drop guard does nothing. When the test body's stack unwinds
and the slices drop, the runtime's `cuMemFreeAsync` is queued on
the alloc stream. The kernel may not have run yet, or may still
be running, on a different stream.

## Why the v0.6 allocator is sound and still cannot rescue this

`AsyncCudaResource` correctly stream-orders `cuMemAllocAsync` /
`cuMemFreeAsync` against the *alloc_stream* recorded in the
`DeviceBlock`. This is proven by:

* A1 parallel allocator stress (8 threads × 64 iters): 1/1 PASS
* A2 stream-lifetime contract (alloc/queue-write/free/realloc no host sync): 2/2 PASS
* A2-via-runtime through the facade: 1/1 PASS
* `test_async_reap_race` (concurrent `deallocate` racing `reap_pending`): 2/2 PASS
* The merged allocator PR's A3: 20/20 PASS at `--test-threads=8`

These tests exercise the allocator's stream-ordering between
operations the **resource itself** queues. They do **not** test
the case where a *kernel* is launched on a different stream and
is still in flight when the slice drops — because the resource
trait has no way to express that dependency.

The prototype submits its launches via cudarc's per-device
default stream (the one returned by `device.inner().stream()`)
and via `htod_sync_copy_into` / `device.synchronize()`. It then
drops `TrackedCudaSlice<T>` instances whose `alloc_stream` is
`StreamId::DEFAULT` (which resolves to the runtime's default
stream slot — the same physical stream as `device.inner().stream()`
in the singleton case, but a *separate* stream in the per-test
runtime case where each test's `with_resource` wraps a new pool).
There is no event coupling between the launch and the queued
free, regardless of which fixture mode is active.

That is why **all three modes fail** identically. The shared
runtime mode fails *worse* because correctly tracking live blocks
across threads exposes more invalidly-used pointers as outright
SIGSEGV instead of silent corruption.

## Required fix: stream/lifetime-safety PR (must land before any join retry)

This is **not** a fix to the prototype's join code. It is a
safety layer the GPU layer needs regardless of the prototype.
Once it lands, the prototype's failures should be reassessed; if
they persist, *then* there is a real join-semantics bug to chase.

Scope:

1. **Provider/Executor execution-stream abstraction.**
   Add a typed handle representing "the stream this provider
   submits work on" — distinct from the device default stream.
   Every operator (filter, compact, dedup, scan, sort, hash join,
   etc.) takes the execution stream as an argument and submits
   all of: cuMemAllocAsync, htod copies, memset, kernel launches,
   dtoh copies, cuMemFreeAsync, on that stream. Kernels that
   genuinely must run in parallel on a non-execution stream
   take an explicit event-wait dependency from the execution
   stream and record an event back on completion.

2. **xlog launch builder with access modes.**
   Replace the raw `Vec<*mut c_void>` pattern with an xlog-side
   builder (e.g., `LaunchBuilder::new(func, cfg).read(slice).write(other_slice).run(stream)`).
   The builder records `read` / `write` / `read_write` modes per
   buffer, captures the raw pointer for the kernel, AND keeps the
   `CudaSlice` references alive (or records explicit events) so
   subsequent free is correctly ordered. cudarc's typed
   `LaunchArgs` already does this — the xlog builder either
   delegates to that path, or implements equivalent
   event-recording manually for the cases cudarc cannot express.

3. **Runtime-backed `TrackedCudaSlice` waits on the real
   last-use event before deallocate.**
   Currently `Backing::Runtime` drop calls `runtime.deallocate(block)`,
   which queues `cuMemFreeAsync` on `block.alloc_stream`. That
   stream is whatever the runtime resolved at allocation time —
   usually the runtime's `StreamId::DEFAULT`. The actual last
   kernel using the memory may have been launched on a *different*
   stream (the provider's execution stream). The free must wait on
   the launch stream's last-use event, recorded at launch time by
   the launch builder above. Concretely: store the last-write
   event on the slice; the runtime's deallocate enqueues a
   `cuStreamWaitEvent(alloc_stream, last_use_event)` immediately
   before `cuMemFreeAsync`.

4. **Failing tests that gate the fix.**
   Add tests that drop runtime-backed buffers immediately after
   queued raw launches *without* a host sync, then immediately
   force pool reuse on a different stream under parallel threads.
   The tests must fail without the safety layer (currently they
   already fail in real_world_tests; we want a tighter, isolated
   reproducer that runs in xlog-cuda's lib tests and can drive a
   clear pass/fail verdict on the safety layer in isolation).

After (1)–(4) land and pass, **then** rebase the binary-join
prototype on top. Its current failures (`mask len > row cap`,
wrong arithmetic rows, missing recursive derivations) are
**symptoms of unsafe GPU lifetime ordering**, not independent
join-semantics bugs, and should not be diagnosed further until
the safety layer holds.

  Documents the stream-ordering requirement for `cudaMallocAsync` /
  `cudaFreeAsync` and the event mechanism for cross-stream
  dependencies.
* NVIDIA CUDA C++ Programming Guide — *Default Stream*. Covers
  per-thread vs per-context default-stream semantics relevant to
  cudarc's `device.inner().stream()` behavior under multi-thread
  use.
* RAPIDS RMM — `device_memory_resource` and `cuda_async_memory_resource`
  documentation. The model the v0.6 allocator mirrors; explicit
  about the caller's responsibility to event-coordinate cross-stream
  reuse.
* cudarc 0.19 source — `LaunchArgs` / `SyncOnDrop` / `CudaSlice`
  drop semantics. Confirms cudarc's safe path records read/write
  events and waits on them before free.
