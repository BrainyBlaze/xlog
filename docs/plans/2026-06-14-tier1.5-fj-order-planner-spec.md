# Tier-1.5 — Free-Join prefix-aware order planner (decline-or-reorder)

Date: 2026-06-14. Branch `feat/factorized-finalize` (from main `e23b70a9`).
Status: **DESIGN — implementation gated on `@human`'s explicit GO** (per "proceed to
finalize after my approval"). Converged with `@dts-dlm-main` in #xlog; this doc captures the
agreed scope so the GO/no-go decision is made against a concrete plan, and so implementation
on GO is mechanical.

## 0. Why this exists (the decider verdict)

Tier-1's `d2_skew_order_decider` (`test_free_join_e2e.rs`) returned the scope-changing result:
on an adversarial blow-up chain `q(A,E) :- e1(A,B), e2(B,C), e3(C,D), e4(D,E)` (prefix
materializes N²=10 000 rows; result = 1 row), **fixed-order FJ peaks 746 KB vs the binary
fallback's 243 KB = 3.07× worse**. The Tier-1 fail-open `factorized_loss_veto` does NOT catch
it — the inputs are large, so the veto (which only fires on a *provably small* join) correctly
declines to fire, and FJ dispatches with the bad traversal order.

Root: the general multiway promoter preserves traversal order and sets `var_order = None`
(see the provenance-gate comment at `wcoj_dispatch.rs:1765`). FJ's prefix constraint then
forces materializing the large intermediate; the binary planner is free to reorder to exploit
the selective tail. This is an **order/planning gap, not a threshold bug** — independently
confirmed by `@dts-dlm-main`.

This is the reason the research program is **not yet fully closed**: a 3.07× dispatched loss
would sit on `main`. Tier-1.5 closes it without the breadth of a full W2.x NDV/heat planner.

## 1. Scope (bounded; beyond approved Tier-1)

Add a **prefix-aware, cardinality-greedy, connected reorder with projection remapping** to
`try_dispatch_free_join`, with a **fail-CLOSED decline to binary** when no prefix-compatible
order is estimated to beat the binary fallback. This is a real (bounded) planner addition —
flagged as genuine scope beyond Tier-1, hence the GO gate.

Explicitly **out of scope**: NDV/selectivity histograms, key-heat statistics, join-graph
treewidth analysis (full W2.6). Tier-1.5 uses *only* the per-relation cardinalities the Tier-1
veto already reads — no new statistics surface.

## 2. Placement — DISPATCH, not the promoter

The reorder lives at runtime dispatch (`try_dispatch_free_join`, `wcoj_dispatch.rs:1749`),
**not** the compile-time promoter, because cardinality is runtime state (`self.stats`) and the
promoter runs with `var_order = None`. It reuses the exact context the Tier-1 veto already
builds (`wcoj_dispatch.rs:1819-1842`): `slot_rels`, `build_wcoj_cost_model(&self.config)`,
`WcojDispatchCtx { stats, width, slot_rels, .. }`. (`@dts-dlm-main` and I corrected an earlier
"promoter" wording to "dispatch" for exactly this reason.)

## 3. Algorithm

Insert between the Tier-1 veto (ends `wcoj_dispatch.rs:1842`) and the dense-variable remap
(begins `:1843`). All estimates use the cardinalities from `populated_cards(ctx)`.

1. **Stats-gated entry (fail-open to current behavior).** If `populated_cards(ctx)` is `None`
   (any slot missing stats — e.g. recursive deltas on early iterations), **skip the planner
   entirely** and proceed with traversal order, exactly as today. No statless regression; this
   preserves every measured D1/D2/D3 win by construction (those fixtures are statless tiny
   programs that must keep firing).

2. **Build the join-variable connectivity graph.** Nodes = input atoms; an edge connects two
   atoms that share at least one join variable. The shared-variable relation is already
   computed downstream as `atom_vars` / `class_to_var` (`wcoj_dispatch.rs:1844-1865`); hoist
   that computation (or a lightweight pre-pass of it) above the planner so connectivity is
   available. The body is connected by construction for a valid MultiWayJoin.

3. **Cardinality-greedy connected order.** Produce a permutation π over input slots:
   - Leader = the atom with the **smallest** cardinality (`populated_cards`).
   - Extend greedily: among atoms **connected to the current prefix** (share a join var with
     some already-placed atom — the prefix-compatible constraint FJ requires), append the one
     minimizing an estimated running prefix cardinality. Use a conservative product estimate
     bounded by the binary-join cardinality bound (no NDV: assume worst-case fan-out capped by
     the partner relation size — deliberately pessimistic so we never *over*-promote FJ).
   - This is O(n²) over n = `inputs.len()` (≥3, small) — trivial cost at dispatch.

4. **Estimate-and-compare (the fail-CLOSED gate).** Compute `est_peak_fj(π)` = max prefix
   cardinality along π, and `est_peak_binary` = the binary fallback's best-reorder peak (same
   greedy estimate, but the binary planner may break the prefix-connected constraint — i.e. its
   estimate is over the unconstrained reorder). **Decline (return `Ok(None)`) iff
   `est_peak_fj(π) > 1.2 × est_peak_binary`.** Otherwise dispatch FJ with order π.
   - The 1.2× slack mirrors the acceptance gate (§5) and absorbs estimate noise.
   - Declining returns control to the binary fallback, which already runs with row parity.

5. **Projection remapping (correctness crux).** If π ≠ identity, the dispatcher can no longer
   assume `output_columns` coincides positionally with concatenated-inputs space (the
   assumption stated at `wcoj_dispatch.rs:1765-1779`). The reorder MUST:
   - permute `bufs`, `slot_vars`, and the per-atom structures by π **together**, and
   - carry π so the emit/projection step that materializes `output_columns` **inverts** π,
     keeping the head columns in fallback projection space.
   This is the part that makes Tier-1.5 a planner addition rather than a one-liner. The
   `d2_skew_order_decider` row-parity assertion (it already checks the FJ result equals the
   binary result row-for-row) is the regression guard for this remap.

## 4. Decline semantics (fail-CLOSED)

Every uncertain branch declines to binary, never forces FJ:
- missing stats → skip planner, keep traversal order (fail-OPEN to current behavior, §3.1);
- planner runs and FJ estimate loses → **decline to binary** (fail-CLOSED, §3.4);
- only fire FJ when an order is estimated within 1.2× of binary.

The adversarial chain's expected outcome is the **decline** path: no prefix-connected order of
a pure chain can avoid the N² intermediate (FJ structurally cannot reorder a chain the way the
binary planner can), so the planner declines and binary runs with parity — satisfying gate
option (2).

## 5. Acceptance gate (strengthened `d2_skew_order_decider`)

The decider PASSES iff EITHER:
1. FJ fires with row parity **AND** peak ≤ 1.2× binary, OR
2. FJ declines **AND** the binary fallback runs with row parity.

PLUS the no-regression bar: **all existing D1/D2/D3 winner fixtures still fire/win** —
- D1 fusion suites (29/29), D2 e2e (6/6), D3 e2e (9/9) unchanged;
- the Tier-1 veto unit tests (5/5) unchanged;
- statless tiny fixtures still fire (§3.1 fail-open).

Measured by extending the existing `blowup_chain_inputs` / `run_with_peak` harness in
`test_free_join_e2e.rs` to assert the disjunction above (today it only *reports* the 3.07×).

## 6. What this does NOT claim

- Not a general planner: it cannot rescue every adversarial body, only decline the ones it
  can't improve. That is the point — no dispatched loss, even if FJ sometimes yields to binary.
- Peak-memory is a functional count (local-measurable); any wall-clock confirmation follows the
  standing RunPod rule. The gate is peak-parity + row-parity, both local-safe.
- D4 remains a parked verified-negative; unaffected.
