# v0.9.0 G090_GPU Workspace Evidence

Date: 2026-05-18

Goal node: `G090_GPU - GPU-Native Runtime And WCOJ Execution`

Branch: `feat/v090-epistemic-solver-semantics`

## Scope

This slice maps `EpistemicGpuPlan` buffer requirements to runtime workspace
layout, allocatable device-buffer handles, device-side workspace reset, bounded
GPU candidate generation, propagation staging, candidate-buffer validation, and
bounded world-view validation/materialization staging, including final-result
flag staging from reduced-output device row-count metadata and final tuple
materialization into a device-resident output buffer. The current slice also
adds accepted v0.7.0 4-cycle plus K5/K6/K7/K8 WCOJ execution evidence through
the production runtime path, K6 G38-B timed helper/histogram reuse, K7/K8 K-clique planner-surface
reuse through runtime preflight, accepted unary/possible/not-possible/not-know/binary/
ternary/multi-membership variable-bound final-row filtering with unary operator, binary
operator, specialized arity-three, and two-literal GPT oracle trace/candidate-index parity,
all-required-membership world-view rejection, independently founded FAEEL
self-`possible` runtime coverage, explicit G91 self-supported `possible`
accepted runtime coverage, specialized arity-three `know fact3(A, B, C)`
accepted runtime coverage, generic arity-N `know fact4(A, B, C, D)` accepted
runtime coverage, operator metrics, binary `not know` absent-key final-row filtering for nonzero-arity
membership, split possible-vs-not-known GPU runtime parity, aggregate
split-batch tracing for zero CPU recomposition and zero per-candidate host
round trips, and a bounded GPT oracle parity check that decodes device rejection
codes into typed GPU reasons.
It does not close `G090_GPU`.
