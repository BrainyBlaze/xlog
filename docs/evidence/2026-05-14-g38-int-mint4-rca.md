## Verdict

M_INT.4 is a real blocker under the current goal-038 contract.

The historical W5.2 baseline is not reproducible on the current machine even
when the old W5.2 branch is rerun directly, so the `+-10%` historical-cell gate
is not currently satisfiable as written. Separately, the integration branch has
an additional WCOJ regression on the 4-cycle cells, especially `4cycle_N1000`
and `4cycle_N2000`.

This RCA first concluded that reintroducing the W5.2-era direct 4-cycle path was
not a safely local correction because W3.3 explicitly retired those direct
symbols. A later HG-preserving mitigation added an E2/E3 prefix plan to the u32
4-cycle HG path instead; see
`docs/evidence/2026-05-14-g38-int-mint4-e2-prefix-attempt.md`.

## Interpretation

1. The W5.2 closure baseline is not reproduced by the old W5.2 branch on this
   machine. Every `5clique` and `pivot5` same-machine ratio is far closer to the
   G38 integration ratio than to the historical baseline.
2. The integration branch adds a distinct 4-cycle WCOJ regression. The hash path
   is broadly comparable between W52 and G38, but G38 GPU time is `4.164x`
   slower on `4cycle_N1000` and `32.022x` slower on `4cycle_N2000`.
3. The `5clique` and `pivot5` cells look primarily like baseline/environment
   drift under this comparison. The 4-cycle cells are code-path-sensitive.

## Root Cause Assessment

M_INT.4 combines two issues:

1. **Baseline reproducibility issue:** the historical W5.2 ratios are not
   reproduced by the old W5.2 branch on the current machine.
2. **Integration 4-cycle regression:** the integrated WCOJ 4-cycle provider now
   routes through the HG work-plan path, and that path is much slower than the
   old direct W5.2 path on the large 4-cycle cells.

The best-supported immediate root cause for the G38-only delta is the W3.3/HG
restructuring of the WCOJ 4-cycle production path, not the W5.2 bench harness
and not the hash-chain comparator.

## Follow-Up Mitigation

`docs/evidence/2026-05-14-g38-int-mint4-e2-prefix-attempt.md` records a
red/green source-audit fix for the G38-only 4-cycle slowdown. The fix keeps the
W3.3 HG kernel symbols and removes the per-work-item linear E2 scan in
`wcoj_4cycle_count_hg_u32` and `wcoj_4cycle_materialize_hg_u32`.

The mitigation improves the large 4-cycle cells substantially:

| Cell | Before GPU ns | After GPU ns |
|---|---:|---:|
| `4cycle_N1000` | 33,939,237 | 1,075,469 |
| `4cycle_N2000` | 261,392,583 | 1,718,195 |

M_INT.4 still fails under the literal historical-ratio window because the
post-fix 4-cycle ratios are above the `+10%` bound while `5clique` and `pivot5`
remain below the `-10%` bound.

## Response Options

1. **Acceptance amendment:** amend M_INT.4 to separate same-machine W5.2
   reproducibility from successor-HG behavior. This now has evidence that the
   G38-only 4-cycle slowdown is fixed, while the literal historical-ratio window
   still is not satisfied.
2. **Further corpus fix:** investigate whether the `5clique` and `pivot5`
   cells can be brought inside the historical ratio window without violating the
   W3.3 HG source-audit locks.
3. **STUCK:** treat G38 as blocked at M_INT.4 under the current contract.

Per S_INT.3, M_INT.5-M_INT.11 remain not run until M_INT.4 is fixed or amended.
