## 1. Постановка задачи

**Цель.** Разрешить телу обучаемого правила `trainable_rule` смешивать нейронные предикаты `nn/4` и
обычные (EDB/derived) отношения в одной клаузе, так чтобы joint-обучение через настоящий движок
(`train_neurosymbolic_program` → `CompiledProgram::forward_backward`) корректно считало вероятность
запроса и дифференцировало её и по сетям, и по весам правил.

**Формально.** Дано правило

```
trainable_rule(r) :: H(x̄) :- N₁(ū₁, ℓ₁), …, Nₘ(ūₘ, ℓₘ), R₁(v̄₁), …, Rₙ(v̄ₙ), B₁, …, Bₖ.
```

где `Nⱼ` — нейронные предикаты (`nn/4`), `Rᵢ` — обычные отношения, `Bₜ` — builtins
(`is`/сравнения/`univ`), `r` — обучаемый вес (через гвард `σ(w_r)`). Сейчас поддерживается только
случай `n = 0`. Требуется поддержать `n > 0` — корректную дифференцируемую семантику запроса
`H` при наличии `Rᵢ` в теле.

**Мотивация.** Паттерн «выученная нейронная оценка, гейченная/структурированная символическими
фактами» — основной в нейро-символике. Флагман-демо «Plasticity & Saliency dILP»:

```
plastic(Edge, strengthen) :-
    saliency(Event, strengthen),     // nn/4: выученная оценка события
    pre_before_post(Event, Edge).    // обычное отношение: структура графа
```

`saliency` без `pre_before_post` не привязана к конкретной связи `Edge` — модель бессмысленна.

**Вне области.** Непрерывная динамика. Дискретизация (биннинг событий в символические факты)
происходит снаружи xlog; движок работает над уже-символическими данными.

## 3. Текущее поведение

### 3.1 Десахаринг и обучение

`crates/pyxlog/python/pyxlog/ilp/neurosymbolic.py`:

- `trainable_rule(id) :: head :- body.` десахарится в обычное правило, защищённое синтетическим
  1-параметрическим нейропредикатом-гвардом `nsr_guard_<id>` (`nsr_w_<id>` — torch-модуль с одним
  параметром-логитом). Вес правила = `σ(logit)` гварда, и его градиент течёт через ту же цепь.
- весь источник компилируется `pyxlog.Program.compile`;
- запросы — `train_head(i)` по индексу строки примера `i`;
- на каждый пример `program.forward_backward(query, target)`; credit assignment через
  `xlog_logic::DifferentiableProofTraceMap`.

### 3.2 Что разрешено в теле

`nn/4`-предикаты и builtins. Запрещено: обычные отношения, негация, эпистемика.

### 3.3 Семантика (проверена тестами)

`P(query) = p_net × σ(w)` с точностью `1e-5`; конъюнкция двух нейроатомов даёт `p₁ × p₂ × σ(w)`;
мутация тела меняет loss; градиенты ненулевые и в сеть, и в вес правила
(`python/tests/test_nn4_dilp_training_surface.py`).

## 4. Проблема — точная формулировка

Обычное отношение в теле `trainable_rule` отклоняется типизированной ошибкой в
`crates/pyxlog/src/neural.rs:1796-1805` (`build_query_signature`):

```rust
if self.neural_registry.get(&body_atom.predicate).is_none() {
    return Err(PyValueError::new_err(format!(
        "Query rule for '{}' references relation '{}/{}' which is not an nn/4 \
         neural predicate; the neural query template path grounds only neural \
         predicates, so this body literal would evaluate over an empty relation",
        pred_name, body_atom.predicate, body_atom.terms.len()
    )));
}
```

Это **сознательный fail-closed**: при текущем построении цепи такое отношение вычислялось бы над
пустым отношением и тихо обнулило бы вероятность. Явная ошибка с именем отношения лучше фальшивого
нуля. Зафиксировано тестом `test_unsupported_body_relation_fails_closed`.

## 5. Корневая причина: архитектура цепи

Ключ — дифференцируемая цепь грунтуется **один раз** на синтетических плейсхолдерах и
переиспользуется на всех примерах подменой весов листьев. Разберём поток целиком.

### 5.1 Сигнатура запроса (`build_query_signature`, `neural.rs:1767-1906`)

Из единственного правила, определяющего `pred/arity`, строится `QuerySignature`:

- каждый нейролитерал тела → `NeuralGroup { info, input_source, … }`;
- `input_source` ∈ `InputSource::QueryArg(head_pos)` (вход берётся из позиции головы) либо
  `ImplicitSlot(slot)` (вход не привязан к голове);
- `QuerySignature::Boolean { groups }` — когда все позиции головы заняты нейровходами (запрос
  фактически ground, супервизия — истинность атома); `QuerySignature::Targeted { target_position,
  groups }` — когда одна позиция головы свободна и пробегает домен меток.

Builtins пропускаются (`_ => continue`, `neural.rs:1791-1793`); негация/эпистемика — ошибка;
**не-нейронное отношение — тот самый fail-closed (§4).**

### 5.2 Шаблон цепи (`generate_template_ast`, `neural.rs:1953-2094`)

Строится синтетическая программа-шаблон:

- для каждой нейрогруппы — annotated disjunction по меткам с **плейсхолдер-константами** в качестве
  ключа входа: `QueryArg` → канонический индекс группы для этой позиции головы, `ImplicitSlot` →
  индекс группы; ключ — `Term::Integer(placeholder)` (`neural.rs:1990-2019`);
- затем `template_program.rules.push(template_rule.clone())` (`neural.rs:2022`);
- грунтованный запрос строится теми же плейсхолдер-константами (`neural.rs:2024-2088`).

**EDB-факты и derived-правила в шаблон не попадают.** Поэтому `Rᵢ(v̄ᵢ)` в теле ссылается на пустое
отношение → коллапс → гвард §4 падает.

### 5.3 Компиляция-однажды и кэш (`forward_backward_complex_tensor`, `neural.rs:1112-1374`)

- сигнатура и шаблон компилируются в d-DNNF/XGCF **один раз на форму** и кэшируются
  (`circuit_cache`, `compile_circuit_for_template`, `neural.rs:1256-1275`);
- на каждый пример: сети группируются по имени, входы стекаются в один батч, единичный
  батч-forward `module(batch)` с трекингом градиента; каждый выход detach-ится и импортируется
  через **DLPack** в `prob_buf: CudaBuffer` (F32, 1 колонка, rows = #меток); рядом создаётся
  `grad_buf` = `zeros_like` для приёма градиента (`neural.rs:1207-1235`);
- выбирается `query_idx` (для `Boolean` = 0; для `Targeted` = позиция метки в `target_domain`);
- вызывается дифф-ядро (GIL отпущен):

```rust
let loss_dev = py.allow_threads(|| {
    cached.program.neural_backward_nll_buffers_with_device_loss(
        &cached.slots, query_idx, &prob_bufs, &mut grad_bufs, cfg, expected,
    )
})?;
```

- loss экспортируется через DLPack в torch-тензор;
- **обратный проход**: по каждому выходу сети `out.backward(grad)`, где `grad` — те самые
  device-буферы, что дифф-ядро заполнило градиентами `dL/dP(label)` (`neural.rs:1362-1371`). Так
  градиент входит в параметры torch.

### 5.4 Модель листа: AD-chain (`neural_backward_nll_buffers_inner`, `exact.rs:931-1140+`)

Каждая «группа» = одно вхождение нейропредиката = набор CNF-переменных (по одной на метку),
образующих AD-chain. Жёсткий инвариант связывает число буферов и число групп:

```rust
if probs.len() != slots.num_groups_usize() {
    return Err(XlogError::Compilation(format!(
        "Neural fast-path error: expected {} groups, got {}",
        slots.num_groups_usize(), probs.len() )));
}
```

Поток на шаге обучения:

1. **Fill** (`exact.rs:1004-1058`): для каждой группы `g` берутся её CNF-переменные
   `slots.group_slot_cnf_var(g)`; ядро `neural_fill_ad_chain_f32` пишет в веса цепи
   `var_log_true`/`var_log_false` значения `log p[label]` и `log(1−p[label])` с клемпами `eps` и
   `min_p`.
2. **Base run** (`exact.rs:1061-1068`): `cache.eval_grads_inplace_fused` считает базовый `logZ` и
   градиенты `dlogZ_base/dp`.
3. **Scatter base** (`exact.rs:1069-1137`): ядро `neural_scatter_ad_chain_grads_f32` раскладывает
   градиенты по `out_grads` для **каждой** группы.
4. **Query run** (`exact.rs:1139+`): `out -= dlogZ_query/dp`; `loss = logZ_base − logZ_query` (NLL
   истинности `query_var`), всё на устройстве.

Итог: вероятность реализуется как взвешенный model count (WMC) над d-DNNF; нейровыходы — это
веса подмножества CNF-переменных, всё остальное (структура тела, конъюнкции, OR голов) запечено в
структуру цепи.

### 5.5 Где именно «теряется» пример

Цепь грунтуется плейсхолдером (напр. `Case → 0`), а идентичность примера `i` несёт только выбор
строки входа (`term_to_input_idx`, `neural.rs:1127`) → веса нейролистьев. Структура цепи
**инвариантна** между примерами. Обычное отношение `Rᵢ(v̄ᵢ)`, чья истинность зависит от примера, в
эту модель не входит: в шаблоне его нет, а механизма «подменить истинность отношения per-example»
не существует.

**Вывод:** ограничение локализовано в построении шаблона и в модели листьев, а не в способности
цепи к джойнам. Это **расширение шаблонного пути**, не новый движок.

## 6. Целевая семантика

Для тела из независимых вероятностных/детерминированных листьев конъюнкция реализуется
произведением весов в WMC. Требуемая семантика запроса для примера `i`:

```
P(H(i)) = σ(w_r) · ∏ⱼ p_netⱼ(i)[ℓⱼ] · ∏ᵢ 𝟙[ Rᵢ выполнено при ключе i ].
```

- нейромножители `p_netⱼ(i)[ℓⱼ]` — как сейчас;
- `σ(w_r)` — гвард правила, как сейчас;
- `𝟙[Rᵢ]` — новый множитель-индикатор: `1`, если отношение `Rᵢ` истинно при ключе примера, иначе
  `0` (вероятность модели обнуляется).

Реализация: `Rᵢ` представляется отдельной CNF-переменной (фиксированный лист), конъюнктится в
запрос, а её per-example вес выставляется в `1` (истинно) или `0` (ложно). Градиент через неё **не
течёт** — ground-факты не дифференцируются. Тонкость: при весе `0` через `log(min_p)` цепь даёт
`≈ε`, а не строгий `0` (см. §10); строгий ноль — форсированием `query`-модели исключить ветви, где
gate-переменная ложна.

## 7. Что движок уже умеет

Вероятностный движок `xlog-prob` уже компилирует детерминированный Datalog в ту же
цепь/provenance, поэтому конъюнкция фиксированного листа — переиспользование существующего:

- детерминированные факты → узлы `const_true` (`crates/xlog-prob/src/provenance.rs:383`);
- джойны тел — единый AND-узел независимо от природы отношения (`provenance.rs:1160`, `eval_rule`):
  `prov2 = builder.and(vec![prov, tuple_prov])`, где у детерминированного кортежа `tuple_prov =
  const_true`;
- несколько голов одного предиката агрегируются через OR (provenance);
- смешанные тела (вероятностное ⋈ derived) уже работают в чистом инференсе
  (`examples/prob/03-recursive-reachability.xlog`);
- тренировочный путь переиспользует **ту же** кэшированную цепь (`exact.rs:1004-1068`).

Пайплайн: семи-наивный provenance (PIR) → CNF-кодирование (все достижимые узлы, включая
`const_true`) → GPU D4→XGCF → кэш цепи на GPU; инференс и обучение делят один `GpuCircuitCache`,
различаясь лишь форсированием evidence vs инъекцией весов + аккумуляцией градиентов.

## 8. Классификация литералов тела

Решение опирается на статическую классификацию каждого литерала тела `trainable_rule`. Пусть
`bound` — множество переменных, связанных до рассмотрения литерала: переменные головы ∪ переменные в
нейро-входных позициях. Тогда:

| Класс | Условие |
|---|---|
| `Neural` | предикат ∈ neural-registry |
| `Builtin` | `Comparison` / `IsExpr` / `Univ` |
| `Negated` | `not p(…)` — не поддержано шаблонным путём |
| `Epistemic` | `know/possible …` — не поддержано шаблонным путём |
| `Gate` | позитивное не-нейронное отношение, **все именованные** переменные ∈ `bound` |
| `UnboundJoin{var}` | позитивное не-нейронное отношение, вводит именованную переменную ∉ `bound` |

Правило безопасности (range-restriction для этого пути): `Gate` ⟺ ни одна позиция не вводит новой
именованной переменной. Анонимный `_` биндит ничего вниз по потоку — разрешён в гейте
(экзистенциальная проекция). Именованная несвязанная переменная — даже если используется нигде
больше — даёт `UnboundJoin` (fail-closed в Stage B); для явного existence-check её переименовывают
в `_`.

- `Gate` → §9 (Stage A): фиксированный лист.
- `UnboundJoin` → §10 (Stage B): требует реальной грунтовки по домену джойна.

Классификация — чистая функция над AST (не зависит от CUDA): принимает литерал, множество `bound` и
предикат-классификатор `is_neural`. `bound` вычисляется вызывающей стороной из головы правила и
нейро-входных позиций.

## 9. Дизайн Stage A — гейты (полная поддержка)

**Область:** обычное отношение делит только уже связанные переменные (head/neural-input), новой
переменной не вводит. Пример: `root_case(Case) :- neural_root(Case, positive), allowed(Case).`

**Упрощатель.** Запрос уже `train_head(i)` по индексу строки → у каждого примера есть символический
ключ `= i` без нового API.

**Механизм** (зеркалит гвард-нейропредикат):

1. **Шаблон.** Каждый `Gate`-литерал эмитится в шаблон как лист с одной «меткой» (annotated
   disjunction из одной альтернативы) на плейсхолдер-ключе — структурно как нейрогруппа, но
   `labels.len() == 1`. Литерал гейта остаётся в правиле, и цепь конъюнктит
   `neural ∧ guard ∧ gate`.
2. **Слоты.** Gate-лист увеличивает `slots.num_groups` → массивы `prob_bufs`/`grad_bufs` должны
   включать его (инвариант §5.4). Чтобы fill отработал, но scatter/backward — нет, gate-группы
   **помечаются отдельно**: их веса заполняются (шаг 1 `inner`), но они исключаются из scatter
   (шаги 3-4) и из torch-backward (нет torch-тензора для гейта). Чистая форма — параллельный набор
   `fixed_slots` + `fixed_weights`, фиксируемый, но никогда не скаттерящийся; это снимает связку
   «fill+scatter по одному и тому же `probs`».
3. **Per-example истинность.** На стороне Python: после `Program.compile` отношение гейта
   материализуется из скомпилированной программы (read-only запрос экстенсии — небольшое новое API,
   если отсутствует); строится булев вектор `gate_truth[i] = (i ∈ extension)`, выровненный по
   строкам примеров. Per-example вес гейт-листа = `1.0`/`0.0` (через `log`/`min_p`), инъектится тем
   же swap-путём, что и нейровеса, но `requires_grad = False`.
4. **Изоляция градиента.** Gate-предикаты исключаются из словарей `neural_parameter_grads` /
   `symbolic_weight_grads`.

**Целевой инвариант теста:** для `allowed(0). allowed(1).` и примеров `0..3` —
`P(root_case(i)) = p_net(i)[positive]·σ(w)` для `i ∈ {0,1}` и `≈0` для `i ∈ {2,3}`; gate-предикат
не получает градиента.

## 10. Дизайн Stage B — джойны (только скоуп)

**Область:** отношение вводит новую экзистенциальную переменную, джойнящуюся с нейропредикатом по
общему домену. Пример (флагман): `plastic(Edge,L) :- saliency(Event,L), pre_before_post(Event,Edge).`

**Почему сложнее.** Плейсхолдер-грунтовка (§5.2) недостаточна: нейропредикат `saliency(Event, …)`
надо грунтовать по **реальному** домену `Event` — несколько листьев на пример вместо одного
плейсхолдерного. Голова `plastic(Edge, …)` агрегирует вклад событий через OR (provenance это уже
делает, §7). Это переделка `generate_template_ast` с «один плейсхолдер на группу» на «реальную
грунтовку по EDB-домену» + маппинг per-example нейровыходов на доменно-ключёванные листья (модель
слотов §5.4 расширяется с «группа = 1 предикат» на «группа = предикат × домен-константа»).

**Условие выполнимости (инвариант кэша).** Сохранить compile-once можно при допущении, что
**реляционная структура одинакова между примерами** (топология `pre_before_post` фиксирована;
меняется только saliency). Тогда цепь компилируется один раз по реальному домену, а per-example
подменяются `N` весов нейролистьев.

**Открытый вопрос.** Примеры со **структурой, меняющейся между примерами** (per-example факты):
требуют либо кэширования цепи по структуре, либо per-example рекомпиляции; стоимость измерить до
выбора. Кандидат на отдельный спек.

**Важно:** правило пластичности — это Stage B, не A. Stage A — необходимый предшественник,
валидирующий механизм инъекции фиксированных листьев, на котором Stage B стоит.

## 12. Риски и краевые случаи

- **`min_p`-клемпинг.** «Загейченная» вероятность `≈ε`, не строго `0`. Ассертить с `abs=1e-6` либо
  спец-кейсить вес `0.0` в жёсткий `false` (форсировать исключение модели), см. §6.
- **Связка fill/scatter.** Текущий код скаттерит по всему `probs`; без разделения нейро/фиксированных
  слотов gate получит «градиент». Разделение слотов обязательно.
- **Нет API экстенсии отношения** в `pyxlog` — мелкое read-only добавление.
- **Рекурсия/деривация в гейте.** Если `Rᵢ` — derived (через правила), его экстенсия берётся из
  семи-наивного инференса той же программы; убедиться, что вычисление экстенсии не зависит от
  обучаемых весов (иначе это уже не константный гейт).
- **Несколько гейтов в одном теле.** Каждый — свой фиксированный лист; конъюнкция в цепи
  тривиальна; per-example вектор истинности на каждый.
- **Скоуп-крип в Stage B.** Классификатор (§8) жёстко стопит `UnboundJoin` отдельной типизированной
  ошибкой; Stage B здесь только спек.
