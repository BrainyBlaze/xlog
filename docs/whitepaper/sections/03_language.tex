\section{The xlog Language}\label{sec:lang}

This section describes the xlog language surface: types, expressions, user-defined functions, modules, and stratified aggregations with constraints. Probabilistic facts (\texttt{p::f.}) and neural predicate declarations (\texttt{nn/k}) are language-level constructs described in \Cref{sec:prob} and \Cref{sec:nesy} respectively.

\subsection{Design Philosophy}\label{sec:lang:philosophy}

xlog is a typed logic programming language designed for GPU execution. Three principles shape its surface:

\textbf{Typed predicates.} Every predicate has a declared signature over a closed set of scalar types (\texttt{u32}, \texttt{u64}, \texttt{i32}, \texttt{i64}, \texttt{f32}, \texttt{f64}, \texttt{bool}, \texttt{symbol}). Type inference verifies argument consistency across rule bodies before any GPU kernel is generated, catching classes of errors that untyped Datalog engines defer to runtime.

\textbf{Stratified semantics as contract.} Negation, aggregation, and recursion interact through a strongly-connected-component analysis on the predicate dependency graph. Programs whose SCCs cross negation or aggregation edges are rejected at compile time. Every accepted program has a well-defined fixpoint and a deterministic stratum order, which the runtime relies on for GPU scheduling.

\textbf{One language for four paradigms.} Probabilistic facts (\texttt{p::f.}), annotated disjunctions, neural predicate declarations (\texttt{nn/k}), and SAT constraints share the same syntactic core with deterministic Datalog. A single parser, type checker, stratifier, and lowerer produces every downstream IR. The distinction between reasoning paradigms lives in the backend, not in the source language.

A minimal xlog program exhibits the core surface:

\begin{lstlisting}[style=xlogstyle]
pred edge(u32, u32).
pred reach(u32, u32).

edge(1, 2). edge(2, 3). edge(3, 4).

reach(X, Y) :- edge(X, Y).
reach(X, Z) :- reach(X, Y), edge(Y, Z).

?- reach(1, N).
\end{lstlisting}

Typed predicate declarations, ground facts, recursive rules, and a query appear in eight lines. The remainder of this section expands each component and adds the extensions that move xlog beyond classical Datalog.

\subsection{Type System}\label{sec:lang:types}

Every predicate is declared with a typed signature via \texttt{pred}. Eight scalar types are supported: \texttt{u32}, \texttt{u64}, \texttt{i32}, \texttt{i64} for fixed-width integers; \texttt{f32} and \texttt{f64} for floats with IEEE 754 semantics; \texttt{bool}; and \texttt{symbol} for dictionary-encoded atoms. String literals in source (\texttt{"alice"}) are terms that map to \texttt{symbol} through the \texttt{SymbolTable} in \texttt{xlog-core}, giving the runtime dense integer identifiers while preserving source-level readability. Users can also introduce named type aliases through \texttt{domain} declarations.

Type inference proceeds in a single pass over the AST. For each rule, variable occurrences across body atoms must agree in type; literal constants are type-checked against their position's declared type; arithmetic expressions propagate types through the operator table. The implementation lives in \texttt{crates/xlog-logic/src/typeinfer.rs}. Errors are emitted with source locations and descriptive messages before lowering begins, so no type error survives into the RIR or reaches the GPU.

A well-typed program:

\begin{lstlisting}[style=xlogstyle]
pred node(u32, symbol).
pred connected(u32, u32).
pred bridge(u32, u32).

node(1, "alice"). node(2, "bob").
connected(1, 2).

bridge(A, B) :- connected(A, B), node(A, _), node(B, _).
\end{lstlisting}

Changing \texttt{bridge}'s declaration to \texttt{pred bridge(symbol, u32).} would cause inference to report that the head argument \texttt{A} is constrained at \texttt{symbol} (head) and \texttt{u32} (through \texttt{connected}), rejecting the program at compile time.

\subsection{Expressions and Arithmetic}\label{sec:lang:arith}

Rule bodies may contain arithmetic expressions, built-in function calls, and comparison filters. Arithmetic bindings use the \texttt{is} operator following Prolog convention: \texttt{D is X + Y} computes \texttt{X + Y} and binds the result to \texttt{D}. The right-hand side supports the standard arithmetic operators (\texttt{+}, \texttt{-}, \texttt{*}, \texttt{/}, \texttt{\%}), the built-ins \texttt{abs}, \texttt{min}, \texttt{max}, \texttt{pow}, and \texttt{cast}, and calls to user-defined functions (\Cref{sec:lang:udf}). Comparison literals (\texttt{<}, \texttt{<=}, \texttt{=}, \texttt{==}, \texttt{!=}, \texttt{>=}, \texttt{>}) compile to \texttt{Filter} RIR nodes and push down below joins where profitable.

Float arithmetic follows IEEE 754 with explicit NaN and $\pm\infty$ handling: \texttt{NaN != NaN}, and ordering predicates place \texttt{NaN} outside any other value rather than raising errors. This matters for aggregations, where a single unguarded NaN would contaminate downstream GPU computations.

A pairwise-proximity query illustrates expression use:

\begin{lstlisting}[style=xlogstyle]
pred point(u32, f64, f64).
pred close(u32, u32).

point(1, 0.0, 0.0). point(2, 0.5, 0.5). point(3, 5.0, 5.0).

close(A, B) :-
  point(A, Xa, Ya),
  point(B, Xb, Yb),
  A < B,
  D is (Xa - Xb) * (Xa - Xb) + (Ya - Yb) * (Ya - Yb),
  D < 1.0.
\end{lstlisting}

The rule compiles to a self-join of \texttt{point} with an intermediate \texttt{Project} that materializes \texttt{D}, followed by a \texttt{Filter} on \texttt{D < 1.0}. The optimizer relocates the cheap comparison \texttt{A < B} ahead of the join when profitable, reducing intermediate cardinality.

\subsection{User-Defined Functions}\label{sec:lang:udf}

xlog extends Datalog with user-defined functions (UDFs): named, typed, side-effect-free computations usable inside rule bodies and expressions. A UDF is introduced with the \texttt{func} keyword; parameters carry optional type annotations, and the return type is declared after a \texttt{->}. Bodies are arithmetic expressions (optionally with conditional \texttt{if/then/else}) or, for more involved logic, a predicate-based form \texttt{V :- body.} that binds the return value by unifying with a relation.

UDFs are pure: they take scalar arguments and return scalar values, and do not recurse through relations. Scalar self-recursion with a base case is permitted; the function registry analyzes each body for convergence and rejects diverging recursion at compile time. Calls to UDFs appear wherever arithmetic expressions do, including as the right-hand side of an \texttt{is}-expression. The implementation lives in \texttt{crates/xlog-logic/src/function.rs}. Before lowering, an inlining pass (\texttt{crates/xlog-logic/src/expand.rs}) substitutes every UDF call by the body's expression tree, so the RIR sees only composed \texttt{Expr} nodes (\texttt{Add}, \texttt{Mul}, \texttt{Conditional}, and so on) that the \texttt{CudaKernelProvider}'s arithmetic module compiles into GPU expression kernels alongside the language built-ins. Because UDFs are pure and inlined, they participate transparently in the same predicate-pushdown and common-subexpression-elimination passes the optimizer already applies to built-in arithmetic.

Most classical Datalog dialects have no UDF analogue: functors are either absent or restricted to uninterpreted term construction. xlog UDFs close this gap without breaking monotonicity. Because bodies are pure scalar expressions, a UDF call in a recursive rule does not enlarge the set of derivable facts beyond what the surrounding Datalog rule already derives.

A UDF for score normalization:

\begin{lstlisting}[style=xlogstyle]
pred raw_score(u32, f64).
pred norm_score(u32, f64).

func clamp01(X: f64) -> f64 =
  if X < 0.0 then 0.0
  else if X > 1.0 then 1.0
  else X.

raw_score(1, -0.3). raw_score(2, 0.7). raw_score(3, 1.4).

norm_score(Id, N) :- raw_score(Id, R), N is clamp01(R).
\end{lstlisting}

The \texttt{clamp01} UDF lowers to a conditional expression embedded in the \texttt{Project} computing \texttt{norm\_score}'s second column; no host dispatch occurs during evaluation.

\subsection{Modules and Imports}\label{sec:lang:modules}

xlog programs decompose into modules. A module is a \texttt{.xlog} file located on the module search path, which the \texttt{xlog} CLI controls via the \texttt{--module-path} flag. The \texttt{use} directive loads a named module (e.g., \texttt{use graph.}) or selectively imports names from it (\texttt{use utils/math::\{abs, clamp\}.}). Module paths use \texttt{/} as a separator and mirror the on-disk directory structure. Declarations and functions carry an optional \texttt{private} modifier to hide them from importers.

Name resolution runs before type inference. The resolver in \texttt{crates/xlog-logic/src/resolver.rs} (469 lines) traverses the import graph, detects cycles, merges symbol tables, and surfaces a single logical program whose remaining compilation stages proceed as if the user had written all rules inline. Imports are transitive; attempts to reach a \texttt{private} name yield a structured \texttt{ModuleError::PrivatePredicate} diagnostic rather than a silent shadowing.

A two-module program where a top-level entry imports a graph-reachability library:

\begin{lstlisting}[style=xlogstyle]
// library: graph_lib.xlog
pred edge(u32, u32).
pred reach(u32, u32).

edge(1, 2). edge(2, 3). edge(3, 4).

reach(X, Y) :- edge(X, Y).
reach(X, Z) :- reach(X, Y), edge(Y, Z).
\end{lstlisting}

\begin{lstlisting}[style=xlogstyle]
// entry: graph_main.xlog
use graph_lib.

?- reach(1, N).
\end{lstlisting}

The resolver merges the two files before stratification, so the library's recursive \texttt{reach} rules participate in exactly the same SCC analysis and GPU semi-naive evaluation as if they had been written inline.

\subsection{Aggregations and Constraints}\label{sec:lang:agg}
