\section{Architecture}\label{sec:arch}

\subsection{System Overview}

xlog's current workspace contains 15 Rust crates organized around a five-tier product/runtime hierarchy. \Cref{fig:arch} summarizes the primary dependency spine and certification surface; two support crates discussed below (\texttt{xlog-induce} and \texttt{xlog-integration}) are omitted from the diagram for readability. The architecture still enforces strict layering: no crate depends on a crate in its own tier or a higher tier.

  \centering
  \includegraphics[width=0.95\linewidth]{architecture.pdf}
  \caption{xlog crate hierarchy. Tier 0 is the leaf foundation; each higher tier depends only on strictly lower tiers.}
  \label{fig:arch}
\end{figure}

\textbf{Tier 0} contains \texttt{xlog-core}, the leaf crate that defines shared scalar types (\texttt{ScalarType}, \texttt{Schema}, \texttt{AggOp}), the \texttt{KernelProvider} trait, error types, memory budgets, runtime configuration, and a bidirectional \texttt{SymbolTable} for dictionary-encoded strings. Every other crate in the workspace depends on \texttt{xlog-core}.

\textbf{Tier 1} provides domain-specific intermediate representations and service providers that depend only on \texttt{xlog-core}. \texttt{xlog-ir} defines the relational IR node tree (\texttt{RirNode}), expression algebra, and \texttt{ExecutionPlan} structures. \texttt{xlog-cuda} wraps CUDA device management via cudarc~\cite{cudarc}, stages CUDA kernel modules (cubin-first with portable PTX fallback), implements the \texttt{CudaKernelProvider} with modular submodules (relational, filter, group-by, arithmetic, I/O, ILP, probabilistic, transfer, kernel loading), and exposes Arrow IPC/C Data and DLPack interoperability. \texttt{xlog-stats} provides \texttt{StatsManager} for compiler feedback and runtime tracking. \texttt{xlog-neural} defines the \texttt{NetworkRegistry}, \texttt{NetworkHandle}, \texttt{TensorSourceRegistry}, \texttt{NeuralBridge}, and \texttt{BatchCollector} types for neural-symbolic integration, with an optional PyO3~\cite{pyo3} feature gate for Python interop.

\textbf{Tier 2} implements the three core subsystems. \texttt{xlog-logic} implements the xlog language frontend (see \Cref{sec:lang}): PEG parser (pest), SCC-based stratifier, AST-to-RIR lowerer, and cost-aware optimizer. An internal rule-rewriting pass in \texttt{expand.rs} (993 lines) normalizes repeated patterns before stratification; this pass is invisible to user programs. xlog programs may also carry \texttt{\#pragma} directives (e.g., \texttt{prob\_engine}, \texttt{prob\_cache}, \texttt{max\_recursion\_depth}) that influence compiler behavior; pragmas parse into dedicated AST nodes consumed by the compiler before lowering. Its production dependencies are \texttt{xlog-core}, \texttt{xlog-ir}, and \texttt{xlog-stats}. \texttt{xlog-runtime} provides the \texttt{Executor} (modular across delta computation, expression evaluation, join caching, node dispatch, recursive evaluation, and rewriting), versioned \texttt{RelationStore}, profiling, and query statistics. It depends on \texttt{xlog-core}, \texttt{xlog-ir}, \texttt{xlog-cuda}, and \texttt{xlog-stats}. \texttt{xlog-solve} implements GPU CDCL verification (complete SAT/UNSAT with on-GPU model and proof validation) and continuous local search for SAT/MaxSAT. It depends on \texttt{xlog-core} and \texttt{xlog-cuda}.

\textbf{Tier 3} composes Tier 2 subsystems into integrated reasoning pipelines. \texttt{xlog-gpu} provides a high-level deterministic execution API with input/output buffer management; it depends on \texttt{xlog-core}, \texttt{xlog-cuda}, \texttt{xlog-ir}, \texttt{xlog-logic}, and \texttt{xlog-runtime}. \texttt{xlog-prob} implements the full probabilistic tier---provenance tracking, PIR-to-CNF encoding, GPU-native D4 compilation, XGCF circuit construction, exact inference, Monte Carlo sampling, and circuit caching---depending on \texttt{xlog-core}, \texttt{xlog-cuda}, \texttt{xlog-ir}, \texttt{xlog-logic}, \texttt{xlog-runtime}, and \texttt{xlog-solve}. \texttt{xlog-induce} is a support subsystem for bounded exact induction on top of the runtime/CUDA stack; it depends on \texttt{xlog-core}, \texttt{xlog-cuda}, and \texttt{xlog-runtime}, and is consumed by the Python ILP surface.

\textbf{Tier 4} presents user-facing interfaces. \texttt{pyxlog} is a PyO3 extension module that bridges nine internal crates (\texttt{xlog-core}, \texttt{xlog-cuda}, \texttt{xlog-gpu}, \texttt{xlog-prob}, \texttt{xlog-logic}, \texttt{xlog-neural}, \texttt{xlog-runtime}, \texttt{xlog-ir}, \texttt{xlog-induce}) to Python, exposing DLPack-first evaluation, training APIs, and ILP/dILP trainers. \texttt{xlog-cli} is the published Cargo package that provides the \texttt{xlog} command-line binary for deterministic and probabilistic execution with Arrow IPC I/O.

Outside the product/runtime stack, \texttt{xlog-cuda-tests} is an unpublished certification suite for GPU release gating, and \texttt{xlog-integration} hosts workspace-only cross-crate executor tests and integration binaries.

\subsection{Compilation Pipeline}

The compilation pipeline transforms Datalog source text into a GPU-executable plan through five stages: parsing, stratification, lowering, optimization, and plan assembly.

\textbf{Parsing.} The PEG parser in \texttt{crates/xlog-logic/src/parser.rs} uses the pest parser generator with a grammar defined in \texttt{grammar.pest}. It produces an abstract syntax tree (AST) containing \texttt{Program}, \texttt{Rule}, \texttt{Atom}, \texttt{Term}, \texttt{BodyLiteral}, and related nodes defined in \texttt{crates/xlog-logic/src/ast.rs}. The parser handles the full xlog language surface (\Cref{sec:lang}), including probabilistic facts, annotated disjunctions, neural predicate declarations, user-defined functions, and module imports.

\textbf{Stratification.} The stratifier in \texttt{crates/xlog-logic/src/stratify.rs} builds a predicate dependency graph with three edge types---positive, negative, and aggregate---then computes strongly connected components (SCCs) using Tarjan's algorithm. If any SCC contains a negative or aggregate edge, the program is rejected with a diagnostic error: stratified negation and stratified aggregation require that no recursion passes through negation or aggregation. The output is an ordered sequence of strata, each containing one or more SCCs that can be evaluated before any stratum that depends on them.

\textbf{Lowering.} The \texttt{Lowerer} in \texttt{crates/xlog-logic/src/lower.rs} translates each AST rule into a \texttt{RirNode} tree defined in \texttt{crates/xlog-ir/src/rir.rs}. Body literals become \texttt{Scan} nodes; variable bindings across atoms produce \texttt{Join} nodes with key columns derived from shared variable positions; negated literals become \texttt{Anti} joins; filters and arithmetic constraints become \texttt{Filter} nodes with \texttt{Expr} predicates (supporting comparison, Boolean connectives, arithmetic, built-in functions, and conditional expressions). The lowerer also emits \texttt{Fixpoint} nodes that wrap recursive SCC bodies for semi-naive iteration, and \texttt{GroupBy} nodes for stratified aggregation.

\textbf{Optimization.} The optimizer in \texttt{crates/xlog-logic/src/optimizer.rs} applies cost-aware transformations using runtime statistics from \texttt{xlog-stats::StatsManager}. It performs predicate pushdown---relocating \texttt{Filter} nodes below \texttt{Project} and \texttt{Join} nodes to reduce intermediate result sizes---and cost-based join ordering via dynamic programming. For queries involving more relations than a configurable threshold (\texttt{dp\_threshold}, default 10), the optimizer falls back to a greedy heuristic to avoid exponential planning time. Cost estimates account for expected row counts, GPU memory consumption, and data transfer volume.

\textbf{Plan assembly.} The final output is an \texttt{ExecutionPlan} (defined in \texttt{crates/xlog-ir/src/plan.rs}) containing SCCs in dependency order, strata for negation ordering, and compiled rule trees grouped by SCC. The \texttt{xlog-runtime} executor interprets this plan by dispatching each \texttt{RirNode} to the appropriate CUDA kernel via the \texttt{CudaKernelProvider}.

\subsection{GPU Residency Model}

xlog enforces a hard guarantee: all runtime semantic state resides in GPU device memory, or the system returns a deterministic error. There is no silent out-of-core spilling, and there is no implicit CPU fallback. If a workload exceeds the GPU memory budget, xlog raises a \texttt{RESOURCE\_EXHAUSTED} error with diagnostic information rather than degrading to host memory transparently.

The residency contract extends to production query paths, where the system targets zero device-to-host (D2H) transfers. The \texttt{CudaKernelProvider} in \texttt{crates/xlog-cuda/src/provider/mod.rs} implements byte-level transfer accounting through a \texttt{HostTransferStats} structure that tracks \texttt{dtoh\_bytes} and \texttt{htod\_bytes} via atomic counters. Callers invoke \texttt{reset\_host\_transfer\_stats()} before a performance-critical section and \texttt{host\_transfer\_stats()} afterward to obtain a snapshot. A separate \texttt{d2h\_transfer\_count} atomic counter increments once per \texttt{download\_column\_*} call, enabling assertions---particularly in the ILP trainer---that no column downloads occurred during a device-resident computation. The \texttt{reset\_d2h\_transfer\_count()} method zeroes this counter for bracketed verification.

This design matters for performance. PCIe~4.0 $\times 16$ delivers roughly 32\,GB/s peak bidirectional bandwidth, while GPU HBM provides 1--3\,TB/s depending on the device. A single unnecessary D2H round-trip for a moderate relation (e.g., 10M tuples at 16 bytes each, roughly 160\,MB) costs approximately 5\,ms of bus transfer latency---enough to dominate a semi-naive iteration step that completes in microseconds on-device. By keeping fact stores, delta relations, circuit node values, solver state, and gradient buffers exclusively on the GPU, xlog eliminates this class of bottleneck. Host involvement is restricted to orchestration (launching kernels, managing plan execution order), I/O (reading source files, writing final results), and compilation (parsing, stratification, lowering, and optimization all run on the CPU before execution begins).

The \texttt{MemoryBudget} configuration in \texttt{xlog-core} already carries an \texttt{allow\_ooc} flag, but current \texttt{main} treats it as reserved configuration rather than an implemented spill path. In today's runtime there is no silent out-of-core execution: workloads that exceed the GPU memory budget fail deterministically with \texttt{RESOURCE\_EXHAUSTED}.

\subsection{IR Stack}

xlog uses a layered intermediate representation stack designed for extensibility across reasoning paradigms. Each IR level serves a distinct role in the compilation and evaluation pipeline.

\textbf{AST (Abstract Syntax Tree).} The parser produces a concrete syntax tree that preserves source-level structure: rules, atoms, terms, probabilistic annotations, neural predicate declarations, and directives. The AST lives in \texttt{crates/xlog-logic/src/ast.rs} and carries no execution semantics.

\textbf{RIR (Relational IR).} The lowerer transforms the AST into an algebraic tree of \texttt{RirNode} variants defined in \texttt{crates/xlog-ir/src/rir.rs}. The key variants are: \texttt{Scan} (base relation access), \texttt{Filter} (predicate evaluation), \texttt{Project} (column selection and computation via \texttt{ProjectExpr}), \texttt{Join} (inner, left-outer, semi, and anti joins with explicit key columns), \texttt{GroupBy} (stratified aggregation), \texttt{Union}, \texttt{Distinct}, \texttt{Diff} (set difference for semi-naive delta computation), and \texttt{Fixpoint} (recursive SCC wrapper). RIR nodes carry metadata including estimated cardinality ranges, memory peak estimates, skew hints for join partitioning, and incremental update semantics (delta vs.\ full materialization).

\textbf{PIR (Provenance IR).} For probabilistic programs, \texttt{xlog-prob} constructs a \texttt{PirGraph} (defined in \texttt{crates/xlog-prob/src/pir.rs}) whose \texttt{PirNode} variants---\texttt{Lit}, \texttt{NegLit}, \texttt{And}, \texttt{Or}, \texttt{Decision}---represent the weighted Boolean formula derived from provenance tracking over the ground program. PIR captures the probabilistic structure needed for knowledge compilation without encoding execution order.

\textbf{XGCF (xlog GPU Circuit Format).} The final compiled form is a levelized DAG stored in \texttt{crates/xlog-prob/src/xgcf.rs} as an \texttt{Xgcf} structure. Each node has a type (\texttt{Const0}, \texttt{Const1}, \texttt{Lit}, \texttt{And}, \texttt{Or}, \texttt{Decision}), child indices, and variable/literal identifiers. Level offsets enable parallel evaluation: all nodes at the same topological level execute in a single kernel launch. Forward passes compute log-space values; backward passes propagate adjoints for gradient computation. The \texttt{GpuXgcf} structure in \texttt{crates/xlog-prob/src/gpu.rs} holds the device-resident buffers. Multiple circuits can be batched via \texttt{circuit\_offsets} for fused evaluation across neural batch dimensions.

\textbf{Extensibility.} The IR stack was designed to accommodate additional representations as the system grows. The architecture documentation describes two planned IRs: \textbf{EIR} (Epistemic IR) for world-view reasoning with modal operators, split plans, and guess spaces; and \textbf{SIR} (Solver IR) for Boolean satisfiability encoding with CNF clauses, cardinality constraints, weight vectors, and proof policies. Neither EIR nor SIR is implemented in v0.5.0. The layered design ensures that adding a new IR requires implementing a lowering pass from RIR (or from another IR at the same level) and a backend that targets either XGCF or a new GPU-resident evaluation format, without modifying existing IR definitions or evaluation paths.
